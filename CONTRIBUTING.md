⚙️ EduPons Backend Contribution Guidelines (NestJS)
This document provides the complete contribution guidelines and best practices for working on the EduPons backend services built using NestJS (Node.js/TypeScript).

1. 🌳 Branching & Commit Conventions
   1.1 Branch Naming Convention
   All branches must start with a descriptive prefix indicating the nature of the change:

`feature/` - New modules, endpoints, or significant additions(e.g.,feature/analytics-module)
`fix/` - Bug fixes for existing functionality(e.g., fix/user-password-reset)
`refactor` - Code restructuring without changing external behavior(e.g., refactor/service-layer-abstraction)
`docs/` - Documentation updates (README, Swagger, etc.)(e.g., docs/api-readme-update)
`test/` - Test additions or modifications(e.g., test/auth-guard-flow)
`chore/` - Maintenance tasks (dependencies, build changes)(e.g., chore/update-nestjs-version)

1.2 Commit Message Format

2. 💻 NestJS & TypeScript Code Style

2.1 General Principles
. Strict Typing: Use TypeScript for all new files. Define proper interfaces, DTOs, and explicit types.

. Avoid any: Only use the any type when absolutely necessary, and always include a comment explaining the necessity.

.  Modular Architecture: Follow the NestJS Module structure. Keep modules small and focused on a single domain.

2.2 Layer Separation

. Controllers: Handle HTTP requests, routing, and input validation. Keep business logic minimal

. Services: Contain the core business logic, handle data manipulation, and communicate with the database/repositories.

. Modules: Aggregate controllers, services, and providers.

2.3 File Naming Convention

Module: PascalCase(e.g., UserModule.ts)
Controller: PascalCase + Controller suffix(e.g., CoursesController.ts)
Service: PascalCase + Service suffix(e.g., CoursesService.ts)
DTO: PascalCase + Dto suffix(e.g, CreateCourseDto.ts)

3. Database Migration Guidelines
All schema changes must be managed via database migrations to ensure consistency across environments.

3.1 Migration Naming Convention 🏷️
Follow the format: [Timestamp]-[Action]-[TableName]

Timestamp: Automatically generated by the ORM/CLI (e.g., 20251207232400).

Action: A verb describing the change (e.g., Create, Add, Drop, Rename).

TableName: The primary table being affected (e.g., UserTable, CourseEnrollment).

Action Example Migration Name
Creation ...-CreateUserTable
Adding a Column ...-AddEmailColumnToUserTable
Renaming ...-RenameCourseNameColumn
Indexing ...-AddIndexToUserId

3.2 Modifying Existing Tables (Adding a Column)
Rule: You must never modify a migration file that has already been merged into main or deployed to a shared environment.

To add a column to an existing table:

Generate a new migration specifically for the schema change.

The migration file should be named using the Add...ColumnTo...Table convention (e.g., [timestamp]-AddAgeColumnToStudentProfile).

In the migration's up method, use the appropriate ORM/Database command to add the new column.

In the migration's down method, include the command to drop the new column.

Example Structure (TypeORM/Prisma Equivalent):

// [timestamp]-AddAgeColumnToStudentProfile.ts

export class AddAgeColumnToStudentProfile {
    public async up(queryRunner: QueryRunner): Promise<void> {
        // Add the new 'age' column
        await queryRunner.addColumn('student_profile', new TableColumn({
            name: 'age',
            type: 'int',
            isNullable: true, // or false, depending on requirements
        }));
    }

    public async down(queryRunner: QueryRunner): Promise<void> {
        // Remove the 'age' column
        await queryRunner.dropColumn('student_profile', 'age');
    }
}

4. 🔒 Security & Data Guidelines

    4.1 Configuration and Secrets
        No Hardcoding: Never hardcode sensitive values (like secrets or passwords) directly in the code.

        Environment Variables: All secrets must be loaded from environment variables using the @nestjs/config module.

        Secret Scanning: Ensure .env files and similar secret files are listed in .gitignore.


    4.2 Security Features

        Authorization: Implement all authorization logic using Guards (e.g., JWT validation, Role-Based Access Control (RBAC) checks) and decorators.

        Input Validation: Use DTOs (Data Transfer Objects) with class-validator and ValidationPipe to strictly validate all incoming data payloads, preventing XSS and SQL injection.

5. ✅ Testing & Quality

5.1 Testing Scope
Unit Tests: Services containing business logic must have unit tests, mocking all dependencies.

E2E Tests: All critical API endpoints and flows (authentication, CRUD) must be covered by End-to-End (E2E) tests.

5.2 Pull Request Checklist
Before submitting a PR, ensure all these steps are complete:

[ ] Code follows the style guidelines defined in Section 2.

[ ] All linting and type checks pass.

[ ] All unit and e2e tests pass.

[ ] Migration file is correctly named and includes an up and down method (if applicable).

[ ] PR description clearly explains the changes and their impact.

[ ] API documentation (Swagger) is updated if new endpoints or schemas were introduced.
